## 一、总体架构设计

系统采用分层架构，核心是 **LangGraph 驱动的多智能体协作层**，上下分别对接数据层和展示层：


```
┌─────────────────────────────────────────────────────────────────┐
│                        展示层（前端）                              │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐            │
│  │ Chat UI │  │ K线图表 │  │ 分析报表 │  │ 风控面板 │            │
│  └────┬────┘  └────┬────┘  └────┬────┘  └────┬────┘            │
│       └────────────┴────────────┴────────────┘                  │
│                         │ WebSocket / REST API                   │
├─────────────────────────┼────────────────────────────────────────┤
│                    LangGraph 多智能体协作层                        │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │                 Supervisor（调度中心）                       │  │
│  └─────────────────────────┬─────────────────────────────────┘  │
│          ┌────────────┬────┴─────┬──────────┬──────────┐       │
│          ▼            ▼          ▼          ▼          ▼        │
│    ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────┐ │
│    │ Chatbot │  │研究员   │   │分析师   │  │决策者    │  │风控  │ │
│    │ Agent   │→ │Agent    │→ │Agent    │→ │Agent    │→ │Agent │ │
│    └─────────┘  └────┬────┘  └────┬────┘  └────┬────┘  └──┬──┘ │
│                      └────────────┴────────────┴──────────┘      │
│                             共享状态（State）                     │
│                    SQLDatabaseToolkit + RAG                       │
├────────────────────────────────────────────────────────────────┤
│                        数据服务层                                 │
│  ┌────────────┐  ┌────────────┐  ┌────────────┐  ┌───────────┐ │
│  │ 数据采集    │→ │ 数据清洗   │→ │ 时序数据库  │→ │ 数据API    │ │
│  │ Scheduler  │  │ Pandas     │  │ TimescaleDB│  │ FastAPI   │ │
│  └────────────┘  └────────────┘  └────────────┘  └───────────┘ │
└─────────────────────────────────────────────────────────────────┘
```

**核心设计理念：**

1. **单一数据源（Single Source of Truth）** ：所有 Agent 通过统一的数据库查询接口获取数据，确保分析结果的一致性。这一点至关重要——研究员提取的数据、分析师计算的指标、风控评估的风险敞口，都基于同一份经过清洗和校验的行情数据。
2. **有向图工作流**：LangGraph 将任务组织成图结构，节点代表 Agent 或函数，边定义了执行路径和数据流动方向。系统通过一个集中的 StateGraph 维护整体上下文，存储中间结果和元数据，支持并行执行和条件分支。
3. **Supervisor 模式**：采用监督者（Supervisor）模式，由一个中央监督 Agent 根据当前上下文和任务需求决定调用哪个专业 Agent，控制所有通信流和任务委托。

## 二、LangGraph 多智能体设计

### 2.1 图结构设计

python

```
from langgraph.graph import StateGraph, END
from typing import TypedDict, List, Dict, Any

class InvestmentState(TypedDict):
    """全局共享状态——LangGraph 的核心"""
    messages: List[Dict]           # 对话历史
    user_query: str                # 用户原始问题
    stock_code: str                # 目标股票代码
    stock_name: str                # 股票名称
    
    # 研究员输出
    market_data: Dict              # 行情数据（日线/周线/月线/年线）
    fundamental_data: Dict         # 基本面数据
    news_sentiment: Dict           # 舆情分析
    
    # 分析师输出
    technical_indicators: Dict     # 技术指标（MA/MACD/RSI等）
    analysis_summary: str          # 分析摘要
    
    # 决策者输出
    recommendation: str            # 投资建议（买入/持有/卖出）
    confidence_score: float        # 置信度评分
    
    # 风控输出
    risk_score: float              # 风险评分
    risk_factors: List[str]        # 风险因素列表
    final_decision: str            # 最终决策（含风控建议）

# 构建图
workflow = StateGraph(InvestmentState)

# 添加节点
workflow.add_node("chatbot", chatbot_node)
workflow.add_node("researcher", researcher_node)
workflow.add_node("analyst", analyst_node)
workflow.add_node("decision_maker", decision_node)
workflow.add_node("risk_assessor", risk_node)

# 设置入口
workflow.set_entry_point("chatbot")

# 添加边和条件边
workflow.add_edge("chatbot", "researcher")
workflow.add_edge("researcher", "analyst")
workflow.add_edge("analyst", "decision_maker")
workflow.add_edge("decision_maker", "risk_assessor")
workflow.add_conditional_edges(
    "risk_assessor",
    should_continue,
    {
        "output": "chatbot",  # 返回结果给用户
        "reanalyze": "researcher",  # 数据不足，重新研究
        END: END
    }
)
```

### 2.2 各 Agent 详细设计

**Chatbot Agent（对话入口）**

| 职责           | 关键能力                                                  |
| :------------- | :-------------------------------------------------------- |
| 解析用户意图   | 识别股票代码/名称、查询类型（行情/分析/决策）             |

**研究员 Agent（数据采集与整合）**

| 职责           | 关键能力                                                 | 依赖工具           |
| :------------- | :------------------------------------------------------- | :----------------- |
| 查询行情数据   | 从数据库获取指定周期的K线数据（7日/日线/周线/月线/年线） | SQLDatabaseToolkit |
| 获取基本面数据 | 财务指标、估值数据、行业分类                             | SQLDatabaseToolkit |
| 采集舆情数据   | 新闻、公告、社交媒体情绪                                 | 网络搜索工具 / RAG |


**分析师 Agent（技术分析）**

| 职责         | 关键能力                               | 依赖工具                  |

| 生成分析摘要 | 综合技术面和基本面，输出结构化分析报告 | LLM                       |


**决策者 Agent（投资建议生成）**

| 职责     | 关键能力                       | 依赖工具 |

| 生成建议 | 给出买入/持有/卖出建议及置信度 | LLM      |
| 支撑理由 | 列出关键支撑逻辑               | LLM      |


**风控评估 Agent（风险校验）**

| 职责     | 关键能力                         | 依赖工具     |

| 最终输出 | 生成带风控建议的最终回复         | LLM          |


### 2.3 条件路由设计

python

```
def should_continue(state: InvestmentState) -> str:
    """风控后的条件路由"""
    if state.get("risk_score", 0) > 0.7:
        # 高风险，可能需要重新分析
        return "reanalyze"
    if state.get("final_decision"):
        # 已有完整输出，返回给用户
        return "output"
    return END
```

## 三、数据层设计

### 3.1 数据采集管道

text

```
数据源（多源） → 采集调度器 → 清洗与对齐 → 时序数据库
     │              │              │            │
  ┌──┴──┐      APScheduler     Pandas     TimescaleDB
 AKShare      (cron)         (统一格式)   (PostgreSQL扩展)
 TickFlow
```


### 3.2 数据库选型

| 数据库                | 用途                                 | 理由                                                         |
| :-------------------- | :----------------------------------- | :----------------------------------------------------------- |
| **TimescaleDB**       | 主数据库（K线数据）                  | PostgreSQL 扩展，既支持标准 SQL 又具备时序数据库的高效插入和查询性能，LangChain 的 SQLDatabaseToolkit 可直接对接 |
| **PostgreSQL**        | 元数据存储（股票基本信息、用户会话） | 与 TimescaleDB 同源，管理便捷                                |
| **Redis**             | 缓存（实时行情、热门股票数据）       | 降低数据库压力，加速响应                                     |
| **Chroma / pgvector** | 向量存储（研报、新闻的 RAG 检索）    | 支持语义检索，增强 Agent 的分析深度                          |

对于 K 线数据，TimescaleDB 是理想选择：它将每个 K 线作为一个数据点存储，使用时间作为索引，并采用数据压缩和索引优化策略，显著提升存储和查询性能。

### 3.4 Agent 与数据库的桥接

利用 LangChain 的 **SQLDatabaseToolkit**，Agent 可以直接通过自然语言生成 SQL 查询并执行：

from langchain_community.agent_toolkits import SQLDatabaseToolkit

db = SQLDatabase.from_uri("postgresql://user:pass@localhost/stock_db")
toolkit = SQLDatabaseToolkit(db=db, llm=llm)

# 研究员 Agent 可用的工具列表：

## 七、关键实现思路

### 7.1 数据一致性保障

- 所有 Agent 通过统一的 `SQLDatabaseToolkit` 访问 TimescaleDB，不直接调用外部 API
- 数据采集层每日定时同步，确保所有分析基于同一时间点的数据快照
- State 对象在 Agent 间传递时保持不可变性，每次更新创建新版本，避免并发冲突

### 7.2 Agent 通信与状态管理

在 LangGraph 中，Agent 通过共享的 State 对象进行通信，而不是点对点消息传递。每个 Agent 读取当前 State 作为输入，返回更新后的版本。LangGraph 使用不可变数据结构管理状态，避免竞态条件。

### 7.3 条件路由与动态决策

通过条件边实现工作流的灵活控制：

python

```
workflow.add_conditional_edges(
    "researcher",
    lambda state: "analyst" if state["market_data"] else "chatbot",
    {"analyst": "analyst", "chatbot": "chatbot"}
)
```

### 7.4 技术指标计算

分析师 Agent 调用工具计算技术指标，核心工具设计：

python

```
from langchain.tools import tool
import pandas as pd
import pandas_ta as ta

@tool
def calculate_ma(data: str, period: int = 20) -> str:
    """
    计算移动平均线
    data: JSON 格式的 OHLC 数据
    period: 周期（默认20）
    """
    df = pd.read_json(data)
    df['ma'] = ta.sma(df['close'], length=period)
    return df[['trade_date', 'close', 'ma']].to_json()
```
